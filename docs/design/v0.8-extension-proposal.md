# BPscript v0.8 — Proposition d'extension (sound prototypes + templates catalogue)

**Statut** : proposition de design, non-implémentée. Demande décision utilisateur
avant tout commit code (cf. mémoire `feedback_ask_before_design.md`).

**Auteur** : agent DESIGNER, 2026-06-03.

**Cibles** : combler deux trous de l'AST/EBNF v0.7 identifiés par le consommateur
BPx :

1. **Sound prototypes** — décrire les ~33 propriétés temporelles d'un symbole
   sound (durée, breakTempo, alphaMin/Max, cover/cont/trunc, pivot, etc.)
   pour que `Locate` / `SetLimits` / `Calculate_alpha` puissent fonctionner
   (cf. `BPx/backlog/m4-symbol-config-audit.md`, BPx
   `src/types/soundConfig.ts`).
2. **Templates régime catalogue** — la section `TEMPLATES:` BP3 utilisée en
   `[mode:tem]` (modus tollens / analyse) et déjà documentée en v0.7
   (`AST.md:374-409`, `EBNF.md:189-241`) mais incomplètement spécifiée côté
   sémantique et arrimage avec le `mode:tem`.

## Préambule — état réel vérifié

Avant de proposer, j'ai relu le code et la spec actuels. Trois points
factuels qui contredisent partiellement le brief initial :

- **L'AST v0.7 reconnaît déjà `@templates`.** Cf. `AST.md:34, 374-409` et la
  grammaire test `test/grammars/simpletemplates/scene.bps`, qui contient
  `@templates / [1] /1 ??? ...` traduite vers `TEMPLATES: / [1] *1/1 ___`
  en BP3. L'EBNF la couvre lignes 189-241.
- **Ce qui manque vraiment côté templates** : l'arrimage entre `[mode:tem]`
  (déjà listé dans `EBNF.md:157`) et le rôle de la section `@templates` comme
  **catalogue à essayer** pendant l'analyse inverse (BP3 `Analyze` /
  `MatchTemplate`, cf. `BPx/backlog/m8-port-plan.md:103-117`). La spec actuelle
  décrit le format de chaque entrée mais ne dit pas *quand* le moteur l'utilise.
- **Côté sound prototypes** : rien dans l'AST v0.7. Aucune section ni champ.
  Le contournement actuel est documenté (`LANGUAGE.md:1357` :
  *« contournement actuel (flat alphabet + bol prefix + prototypes -so.) est
  insatisfaisant »*).

Cette proposition cible donc :
- **A** : ajout pur côté sounds (rien en v0.7).
- **B** : *clarification* sémantique de `@templates` + arrimage explicite à
  `[mode:tem]` (la section existe en v0.7, son rôle n'est pas spécifié).

# Partie A — Sound prototypes

## A.1 Contexte BP3 vérifié

Chaque symbole `sound` BP3 a ~33 propriétés indexées par `j` (`Jbol`),
déclarées dans `-BP3decl.h:185-200` (les `p_Xxx` globaux), initialisées par
`ResetPrototype` (`SoundObjects3.c:43-117`), surchargées par
`LoadObjectPrototypes` quand un fichier `-so.NAME` est chargé
(`SaveLoads1.c:920-1080`).

Catégories (cf. `BPx/backlog/m4-symbol-config-audit.md:26-87`) :

1. **Capacités booléennes** (11) — `OkRelocate`, `BreakTempo`,
   `ContBeg/End`, `CoverBeg/End`, `TruncBeg/End`, `FixScale`, `OkExpand`,
   `OkCompress`.
2. **Bornes temporelles** (12, paires valeur+mode ABSOLU/RELATIF) —
   `MaxDelay`, `MaxForward`, `MaxBegGap`, `MaxEndGap`, `MaxCoverBeg/End`,
   `MaxTruncBeg/End`, `BreakTempoMode`.
3. **Durée et résolution** (3) — `Dur`, `Tref`, `Resolution`.
4. **Alpha** (2 utilisés, 4 différés) — `AlphaMin`, `AlphaMax`,
   `AlphaCtrl*`, `RescaleMode`.
5. **Pivot pour dilatation** (3) — `PivType` ∈ [1,7], `PivPos`, `PivMode`.
6. **Période interne** (7) — `PeriodMode`, `BeforePeriod`, `PreRoll`+mode,
   `PostRoll`+mode, `ForceIntegerPeriod`.
7. **Capacités d'altération** (7, défauts seulement en M4) — `OkTransp`,
   `OkArticul`, `OkVolume`, `OkPan`, `OkMap`, `OkVelocity`.

Format historique BP3 = un fichier annexe `-so.NAME` par grammaire, format
positionnel (un nombre par ligne, ordre fixe), cf. `bp3-engine/test-data/-so.dhati`
et le parseur `LoadObjectPrototypes` (`SaveLoads1.c:920-1080`).

**Distinction importante avec `sounds:` actuel** : `@actor sitar sounds:tabla_perc`
en v0.7 désigne le **timbre / synthèse runtime** (lib JSON, ex
`lib/sounds/tabla_perc.json`). C'est consommé par le **dispatcher** au
playback, jamais par le moteur BP3. Les propriétés `-so.` BP3 (durée, alpha,
cover/cont/trunc, etc.) sont consommées par le **moteur** lui-même
(`Locate`, `SetLimits`, `Calculate_alpha`). Les deux mondes ne se croisent pas
sémantiquement et ne doivent pas partager une même clé.

## A.2 Trois syntaxes possibles

### Option A1 — Section dédiée `@sound`

```bpscript
@sound
  ki { dur:500, breakTempo:true, alphaMin:50, alphaMax:200, pivType:5 }
  ge { dur:300, contBeg:true }
  dhin { dur:1000, coverBeg:true, coverEnd:true, maxCoverBeg:50 }
```

Forme : nom du symbole, accolades, paires `clé:valeur` séparées par virgules.
Le bloc s'étend jusqu'à la prochaine directive `@`, le séparateur `-----`,
ou la fin de fichier.

**Avantages** :
- Symétrie complète avec `@templates`, `@actor`, `@cc`, `@mode` — un seul
  pattern « top-level section déclarative », l'utilisateur n'a qu'un modèle
  mental à acquérir.
- Discoverabilité : un `grep @sound` ou un fold-marker UI suffit à inventorier.
- Indépendant des acteurs : un même prototype peut être réutilisé par plusieurs
  acteurs (`@actor tabla1 sounds:tabla_perc`, `@actor tabla2 sounds:tabla_perc`).
- Réutilise les conventions de tokenisation existantes (`{key:value, ...}` est
  déjà reconnu sous d'autres formes — `runtime_qualifier`, `param_pairs`).
- Migration : zéro impact sur les 44 grammaires existantes. Le bloc est
  optionnel.
- Implémentation parser : ajoute un état `INSIDE_SOUND_BLOCK` au tokenizer,
  parse jusqu'à la prochaine `@` / `-----` / EOF. Coût ~50 LOC parser.

**Inconvénients** :
- Une grammaire avec 30 symboles donne un bloc de 30 lignes — verbeux mais
  lisible.
- Pas de portée par acteur : pour qu'un même symbole `Sa` ait deux configs
  selon l'acteur, il faut un mécanisme additionnel (qualifier `sitar.Sa { ... }`,
  cf. dot notation). Peu probable en pratique mais à documenter.

---

### Option A2 — Inclus dans `@alphabet`

```bpscript
@alphabet.tabla
  ki { dur:500, breakTempo:true }
  ge { dur:300 }
```

Le bloc suit la directive `@alphabet.X` et lie les configs au sous-ensemble de
symboles importés par cet alphabet.

**Avantages** :
- Co-location sémantique : le timing d'un symbole est attaché à son alphabet
  d'origine.
- Évite le couplage croisé (un alphabet n'altère pas les symboles d'un autre).

**Inconvénients** :
- **Confusion sémantique majeure** : `@alphabet` charge actuellement un
  fichier `lib/alphabets.json` (un *catalogue* externe de noms). Inliner des
  configs par symbole dans la directive rend l'alphabet partie-fichier
  partie-inline, alors qu'aujourd'hui c'est strictement un identifiant de
  lookup.
- Si plusieurs acteurs partagent le même alphabet mais veulent des timings
  différents, on doit dupliquer la directive — qui n'est pas conçue pour ça
  (`@alphabet.tabla` est unique par scène).
- Casse une convention forte (cf. `design_decisions.md` : « Libraries are
  JSON, not in-language »). Mettre des données par symbole dans
  `@alphabet.X` la mélange à du code.
- Impact migration plus complexe : 44 grammaires utilisent déjà
  `@alphabet.X:transport` — ajouter un bloc à suivre nécessite un parser
  multi-mode.

---

### Option A3 — Fichier annexe (mimétique BP3 `-so.`)

```bpscript
@include sound:tabla.so.bps
```

Le fichier `tabla.so.bps` contient le bloc lui-même (en JSON ou en syntaxe
BPscript).

**Avantages** :
- Garde le `.bps` principal court.
- Réutilise littéralement la convention BP3 (un fichier `-so.X` par grammaire).
- Permet de partager un même catalogue entre plusieurs scènes (`@include`).

**Inconvénients** :
- Casse la règle « source de vérité unique » (cf. `m4-symbol-config-audit.md:255`
  « Le `.bps` reste la source de vérité unique d'une scène musicale, pas de
  fichier annexe »).
- Introduit `@include` (n'existe pas dans la v0.7) — extension de langage
  plus large que le sujet.
- Loader doit gérer la résolution de chemins et le cache.
- Lecture en partage entre éditeurs : il faut deux onglets ouverts pour voir
  un symbole et sa config.

---

### Forme micro-syntaxe : valeurs et modes

Quelle que soit l'option retenue (A1 recommandée — voir D), trois questions
de micro-syntaxe restent ouvertes.

**Modes ABSOLU vs RELATIF**.

BP3 utilise `ABSOLU = -1, RELATIF = 0` (`-BP3.h:297-298`). BPx canonise en
strings `'absolute' | 'relative'` (`soundConfig.ts:60`). Deux écritures
possibles :

- **A.M1** : `delayMode:absolute, maxDelay:500` — deux champs distincts,
  fidèle au mapping BPx.
- **A.M2** : `delay:500ms` / `delay:50%` — suffixe d'unité. Plus court mais
  ambigu (`50%` de quoi ? `dur * dilationratio` côté BP3). Le `%` est déjà
  utilisé en CV ; introduire `ms` n'a pas de précédent BPscript.

**Recommandation A.M1** : deux champs explicites. La conversion BP3 raw
existe déjà côté BPx (`parseModeAbsRel`, `soundConfig.ts:382`).

**Pivot — type 1..7**.

BP3 stocke `PivType` comme entier 1..7 (`SoundObjects3.c:164-167`).
BPx accepte les deux formes (`pivType: number` validé ∈ [1,7],
cf. `soundConfig.ts:233`). Trois écritures :

- **A.P1** : `pivType:5` — entier brut.
- **A.P2** : `pivType:center` — string lisible. Mapping :
  `1=begin, 2=end, 3=begon, 4=endoff, 5=center, 6=centonoff, 7=spec`.
- **A.P3** : les deux acceptés en entrée, normalisés à l'usage.

**Recommandation A.P3** : accepter les deux. Le parser convertit string →
entier 1..7 via une table fixe (déjà documentée `soundConfig.ts:77-86`).
L'AST ne contient que la forme canonique.

**Booléens nus**.

`{ breakTempo, contBeg }` ≡ `{ breakTempo:true, contBeg:true }`.

- **A.B1** : nom nu = `true`. Cohérent avec la convention engine_qualifier
  (`EBNF.md:399` : *« engine_pair = ENGINE_KEY [« : » raw_value] »* ;
  *« flag nu : [destru] »*) et runtime (cf. `[retro]`, `[destru]`).
- **A.B2** : explicite obligatoire `breakTempo:true`. Plus verbeux mais
  uniforme.

**Recommandation A.B1** : accepter le nom nu pour `true`. Cohérent avec
l'existant.

## A.3 Forme AST recommandée pour A1

```typescript
// Extension de Scene (AST.md:20-36)
interface SceneAST {
  ...
  soundPrototypes: SoundPrototypeAST[] | null;  // section @sound, null si absente
}

interface SoundPrototypeAST {
  type: "SoundPrototype";
  symbol: string;                  // nom du symbole : "ki", "ge", "dhin"
  config: Partial<SoundConfigInput>;  // sous-ensemble des ~32 props (BPx soundConfig.ts:194-251)
  line: number;
}
```

Aucun nouveau type à introduire au-delà : `SoundConfigInput` est déjà la
shape négociée côté BPx (`src/types/soundConfig.ts:194-251`). Champs
absents = défaut `ResetPrototype` complété par `mergeWithDefaults`
(`soundConfig.ts:444`).

**Note de design importante** : le champ `config` accepte des modes en
forme string (`'absolute'`) **et** en forme numérique BP3 brute (`-1`).
La forme string est canonique pour BPscript v0.8+ ; la forme numérique
sert à ingérer (futur loader BP3 `-so.` → AST). Le parser BPscript émet
toujours la forme string. (Cf. `soundConfig.ts:382-416`.)

# Partie B — Templates régime catalogue

## B.1 Contexte et état v0.7

La section `@templates` **existe** dans l'AST v0.7 (`AST.md:34, 374-409`)
et dans l'EBNF (`EBNF.md:189-241`), avec un format inspiré de BP3 :

```bpscript
@templates
[1] /1 ???????
[2] /1 ?????????
[3] /1 ($0 ???)($1 )
```

Traduit en BP3 :

```
TEMPLATES:
[1] *1/1 _______
[2] *1/1 _________
[3] *1/1 (@0 ___)(@1 )
```

Test couvrant : `test/grammars/simpletemplates/scene.bps`.

**Ce qui est documenté** : le format de chaque entrée, le mapping
`? → _`, `($N ...) → (@N ...)`, `/N → *N/1`.

**Ce qui n'est pas documenté** (et qui est l'objet de la partie B) :

1. **Quand le moteur utilise ces templates** (sémantique d'exécution) —
   l'`AST.md:406-408` dit seulement *« Si présente, l'encoder émet la
   section TEMPLATES: »* mais n'explique pas le **régime catalogue** :
   en mode `[mode:tem]`, le moteur essaie chaque entrée comme contrainte
   structurelle contre l'item dérivé (BP3 `Analyze` /
   `MatchTemplate`, cf. `m8-port-plan.md:103-117`).
2. **Comment l'utilisateur active le régime** — y a-t-il besoin d'une
   directive globale (`@mode:tem` sur la première sous-grammaire) ou est-ce
   automatique dès que `@templates` est présent ?
3. **Liens avec `$X &X` inline** — l'EBNF distingue actuellement deux
   mécanismes : (i) `$X &X` inline dans les RHS (déjà supporté,
   `EBNF.md:622-645`, régime A « capture/replay pendant la dérivation »
   selon `m8-port-plan.md:99-101`), (ii) `@templates` section (régime B
   « catalogue post-dérivation » selon `m8-port-plan.md:102-117`). La
   spec ne dit pas qu'ils sont distincts.

Le **vrai trou** côté templates n'est donc pas l'AST ni l'EBNF — c'est
**LANGUAGE.md** (la sémantique d'exécution) et un éventuel raffinement de
l'AST pour signaler explicitement les deux régimes.

---

## B.2 Trois syntaxes possibles pour le régime catalogue

Les trois options décrivent **comment l'utilisateur déclare que les
templates sont à utiliser en mode catalogue** (= analyse post-dérivation).

### Option B1 — Mode sur la sous-grammaire suivante (mimétique BP3)

```bpscript
@mode:random
S <> C4 . C5 . D5
S <> C4 . C5 . F#4
S <> $A &A
A <> D5 C5

-----

@mode:tem
@templates
[1] /1 ???????
[2] /1 ?????????
[3] /1 ($0 ???)($1 )
```

Le `@mode:tem` **avant** la section `@templates` signale qu'il s'agit du
régime catalogue. La section `@templates` est juste le payload (la liste
des templates à essayer).

**Avantages** :
- Mimétique BP3 : en BP3, la section `TEMPLATES:` apparaît dans une
  sous-grammaire dédiée séparée par `-----`. Cf. `bp3-engine/test-data/-gr.dhin1`
  ligne 67-78 : la section `TEMPLATES:` suit un séparateur, comme une
  sous-grammaire à part entière.
- Uniforme avec les autres modes : `@mode:random`, `@mode:lin`, `@mode:tem`
  partagent la même grammaire.
- Migration claire des grammaires BP3 historiques.

**Inconvénients** :
- Le `@mode:tem` n'est pas le mode d'une *sous-grammaire de règles*, c'est
  le mode d'usage d'**un catalogue de templates** — c'est un détournement
  léger de la sémantique de `@mode` (modes actuels = stratégies de dérivation
  sur des règles).
- Verbeux : deux directives consécutives.

---

### Option B2 — Marqueur sur la section `@templates`

```bpscript
@templates:catalog
[1] /1 ???????
[2] /1 ?????????

@templates:auto                 // équivalent à `auto` = régime A inline
```

Le suffixe `:X` qualifie le rôle de la section.

- `catalog` (ou `tem`) = régime catalogue, utilisé en analyse inverse,
  matché contre chaque item dérivé.
- `auto` = mode laissé à BP3 (génération automatique des templates pendant
  la production, comportement actuel quand la section est absente).

**Avantages** :
- Une seule directive — auto-suffisant.
- Discoverable : la signification du catalogue est portée par la section
  qui en contient le payload.
- Le suffixe `:X` est déjà la convention `directive_body` (`EBNF.md:62`,
  ex `@routing:studio`).

**Inconvénients** :
- Demande de réserver des sous-clés (`catalog`, `auto`, peut-être
  `inline`).
- Si l'utilisateur ne met pas de sous-clé, faut-il un défaut ? Recommandation :
  défaut = `catalog` quand la section est présente, le seul cas qui ait du
  sens en pratique pour la v0.8.

---

### Option B3 — Marqueur global `@analyze` séparé

```bpscript
@templates
[1] /1 ???????
[2] /1 ?????????

@analyze:templates              // active le régime catalogue
```

Deux directives indépendantes : `@templates` déclare le catalogue,
`@analyze:templates` active l'analyse inverse au playback.

**Avantages** :
- Sépare *contenu* (`@templates`) et *usage* (`@analyze`). Permet
  potentiellement d'autres usages futurs des templates (génération guidée,
  validation, etc.).
- N'altère pas la syntaxe `@templates` existante.

**Inconvénients** :
- Deux directives à coordonner — risque d'erreurs utilisateur
  (`@templates` sans `@analyze`, ou inverse).
- Plus de surface API à documenter et à tester.

---

## B.3 Distinction explicite des deux régimes

Quelle que soit l'option syntaxique B1/B2/B3, il faut **expliciter** dans
la spec la distinction entre les deux mécanismes templates :

- **Régime A — capture/replay** (déjà supporté en v0.7) : `$X ... &X` dans
  les RHS d'une règle, ou `${...}` / `&{...}` sur groupe. Compilé en
  `(=X)` / `(:X)` dans la grammaire BP3, traité par
  `Encode.c:FindMaster/BindSlaves`. La capture/replay opère **pendant** la
  dérivation (cf. `m8-port-plan.md:100-101`).

- **Régime B — catalogue post-dérivation** : section `@templates` avec
  entrées `[N] /scale ???`. Utilisé en `[mode:tem]`. Le moteur essaie
  chaque entrée contre un item dérivé pour reconstruire la dérivation
  qui l'a produit (modus tollens). Cf.
  `m8-port-plan.md:103-117, ProduceItems.c:Analyze`.

Les deux régimes peuvent coexister dans la même scène : `$X &X` dans les
règles + un catalogue `@templates` global. C'est le cas du fichier
`-gr.dhin1` historique.

---

## B.4 Forme AST recommandée

La structure `TemplateEntry` (`AST.md:374-409`) est déjà correcte. Il faut
ajouter un champ qui indique le régime activé :

```typescript
interface SceneAST {
  ...
  templates: TemplateEntry[] | null;       // déjà en v0.7
  templatesMode: 'catalog' | 'auto' | null; // NOUVEAU — null si pas de @templates
}
```

- `'catalog'` : régime catalogue actif, le moteur applique en `[mode:tem]`.
- `'auto'` : compatibilité — la section est présente mais sans usage actif
  (le moteur génère les templates automatiquement pendant la production).
- `null` : pas de section `@templates`.

Aucun changement de structure pour les entrées individuelles. Compatible
ascendant.

# Partie D — Recommandation argumentée

## D.1 Sound prototypes — recommandation : **Option A1 (section dédiée `@sound`)**

**Arguments** :

1. **Cohérence avec le reste de la spec** : `@sound` reproduit le pattern
   des autres top-level sections (`@templates`, `@actor`, `@cc`, `@mode`).
   Un seul modèle mental. Cf. `EBNF.md:14-19` — `scene = { directive | ... |
   declaration | ... }+`.
2. **Lisibilité** : `ki { dur:500, breakTempo:true }` se lit immédiatement
   comme « prototype timing du symbole `ki` ». Pas de confusion avec
   l'aspect timbre/synthèse (qui reste sous `@actor ... sounds:tabla_perc`).
3. **Migration zéro impact** : aucune des 44 grammaires existantes ne
   contient `@sound` aujourd'hui, donc ajouter cette section ne casse rien.
   Liste des grammaires inspectées : voir `test/grammars/` (44 répertoires).
4. **Découplage propre** : `@sound` ne pollue ni `@alphabet` (libs JSON
   immutables) ni `@actor` (binding scale/timbre/transport). C'est une
   nouvelle couche bien définie.
5. **Implémentation parser** : ajoute un seul état de tokenizer
   (`INSIDE_SOUND_BLOCK`), parse le bloc jusqu'au prochain `@` / `-----` /
   EOF, émet un `SoundPrototypeAST` par symbole. ~50 LOC parser, ~30 LOC
   tokenizer. Mineur.
6. **Compatibilité futur GUI** : un widget « Sound prototypes » peut
   éditer le bloc `@sound` indépendamment du reste, sans toucher aux règles
   ni aux acteurs. Section bien isolée.
7. **Source de vérité unique** : tout reste dans le `.bps` — pas de fichier
   annexe à synchroniser (rejet de A3).

**Rejets** :

- **A2** : casserait la sémantique d'`@alphabet` (lib JSON externe → bloc
  inline).
- **A3** : casserait la règle « .bps source unique » de
  `m4-symbol-config-audit.md:255`. Introduit `@include`, hors scope.

**Soumis à l'utilisateur** :

- Le nom de la section : `@sound` (singulier) vs `@sounds` (pluriel) vs
  `@prototypes` ? Recommandation `@sound` par parallélisme avec `@actor`
  (singulier), `@scene`. Mais `@templates` est pluriel… ouvert.
- Le séparateur de champs : virgule (`{ a:1, b:2 }`) vs blanc (`{ a:1 b:2 }`).
  La virgule est cohérente avec `runtime_qualifier` (`EBNF.md:425`,
  `(vel:80, pan:64)`). **Recommandation** : virgule.
- Acceptation booléens nus (`{ breakTempo }` ≡ `{ breakTempo:true }`) ?
  **Recommandation** : oui, cohérent avec `[destru]` (`EBNF.md:399`).

## D.2 Templates catalogue — recommandation : **Option B2 (marqueur sur la section)**

**Arguments** :

1. **Une seule directive** : `@templates:catalog` est auto-suffisant —
   l'utilisateur écrit la section et déclare son rôle en même temps. Pas de
   risque d'inconsistance entre directives séparées (rejet de B3).
2. **Le suffixe `:X` est déjà la convention BPscript** :
   - `EBNF.md:62` — `directive_body = IDENT « : » IDENT`
   - Existants : `@routing.studio`, `@tuning.equal_temperament:western`,
     `@mode:random`.
   `@templates:catalog` suit cette grammaire sans déroger.
3. **Compatibilité ascendante** : `@templates` (sans suffixe) reste
   accepté, traité comme défaut `catalog` (c'est de facto le seul usage qui
   ait un sens en v0.8 ; le mode `auto` est nominalement supporté mais
   redondant avec « pas de section »).
4. **Évite le couplage avec `@mode:tem`** : B1 mélange « mode de
   sous-grammaire de règles » et « régime d'usage du catalogue » dans la
   même directive — sémantiquement distinct.
5. **Migration claire** : `test/grammars/simpletemplates/scene.bps` (le
   seul usage actuel) reste valide ; on lui ajoute juste le suffixe quand
   le régime `catalog` sera implémenté.

**Rejets** :

- **B1** : nécessite que l'utilisateur écrive `@mode:tem` avant
  `@templates`, et cette directive vit dans un autre cadre conceptuel
  (mode de sous-grammaire de règles).
- **B3** : deux directives à coordonner — surface d'erreur double.

**Soumis à l'utilisateur** :

- Nom du marqueur : `:catalog`, `:tem` (mimétique BP3), `:analyze` ?
  **Recommandation** : `:catalog` (descriptif en termes du domaine,
  cf. `feedback_bpx_no_jargon_in_conversation.md`). `tem` est interne BP3,
  pas un mot du domaine musique.
- Faut-il aussi documenter `:auto` ? **Recommandation** : non en v0.8.
  Garder seulement `:catalog` (et `@templates` nu = défaut catalog). Quand
  un cas d'usage `auto` se présentera, on l'ajoutera.
