## Décisions

### 1. Sons (`@sound`) — modèle hybride cascade + sons nommés

**Territoire `@sound`** = uniquement déclaratif. Les affectations à un sujet
(note, alphabet, acteur) se font depuis le territoire d'origine du sujet,
jamais depuis `@sound`.

```bpscript
@sound
  { dur:500, alphaMin:80, alphaMax:120 }     // entrée anonyme = défaut hérité par les sons déclarés
  bell_short { sample:"bell.wav", dur:400 }
  bell_long  { sample:"bell.wav", dur:1200, coverEnd:true }
  drum_kick  { sample:"kick.wav", dur:200, breakTempo:true }
```

Un son décrit à la fois son **timbre** (`sample:` ou `synth:`) et son
**comportement temporel** (durée, alpha, cover/cont/trunc, pivot, période, etc.).
Pas de directive `@synth` séparée — la séparation est dans les champs.

Les 33+ propriétés viennent de l'audit BP3 (cf. `BPx/backlog/m4-symbol-config-audit.md`).
Forme canonique : modes en string (`'absolute' | 'relative'`), pivType accepte
string ou entier 1..7, booléens nus = `true`.

### 2. Affectations note/alphabet/acteur → son

Dans le territoire de chaque élément :

```bpscript
@alphabet.tabla
  notes: Sa Re ga ma Pa dha ni
  *:sound.bell_short                          // défaut alphabet (explicite via *)
  Sa:sound.drum_kick                          // affectation note Sa
  Re:sound.bell_long

@actor sitar
  alphabet.tabla
  tuning.equal_temperament:western
  transport.midi(ch:10)
  *:sound.bell_short                          // défaut acteur
  Sa:sound.drum_kick                          // sitar joue Sa avec drum_kick
```

Et inline dans une règle (override par occurrence) :

```bpscript
S -> Sa(sound.bell_short)                    // override CT level
```

### 3. Cascade — 8 niveaux

Du moins spécifique au plus spécifique. Le moteur résout les propriétés d'un
son joué par un acteur en fusionnant les niveaux **par propriété** (héritage
fin, à la CSS).

| # | Niveau | Source |
|---|---|---|
| 1 | Défaut moteur BP3 | constantes `ResetPrototype` (SoundObjects3.c:43-117) |
| 2 | Défaut anonyme de scène | `@sound { ... }` (entrée sans nom) |
| 3 | Défaut alphabet | `@alphabet.X *:sound.NAME` |
| 4 | Défaut note dans alphabet | `@alphabet.X Y:sound.NAME` |
| 5 | Défaut acteur | `@actor X *:sound.NAME` |
| 6 | Défaut note d'acteur | `@actor X Y:sound.NAME` |
| 7 | Inline sur occurrence | `Y(sound.NAME)` dans RHS |
| 8 | (Réservé) override CV runtime | future v0.9+ |

Chaque niveau peut soit pointer un son nommé (référence), soit donner un bloc
de propriétés anonyme qui s'ajoute à la cascade.

### 4. Templates (`@template`) — singulier, sans suffixe

```bpscript
@template
[1] /1 ???????
[2] /1 ?????????
[3] /1 ($0 ???)($1 )
```

- Renommage `@templates` (v0.7) → `@template` (v0.8). Singulier, aligné avec
  la convention `@actor`, `@sound`, `@alphabet`, etc.
- Pas de suffixe : la section est **toujours** en mode catalogue (utilisée
  en `[mode:tem]` pour l'analyse inverse / modus tollens).
- Sémantique côté moteur : régime B (catalogue post-dérivation), cf.
  `BPx/backlog/m8-port-plan.md:103-117`.

### 5. Conventions de notation cristallisées

| Symbole | Sens | Exemple |
|---|---|---|
| `.` | référence à un élément dans un namespace | `sound.bell_short`, `alphabet.tabla`, `transport.midi` |
| `:` | affectation à un sujet | `Sa:sound.drum_kick`, `*:sound.bell_short` |
| `*` | sujet = défaut (wildcard) | `*:sound.bell_short` |
| `()` | paramètres runtime (héritables) | `Sa(vel:80)`, `transport.midi(ch:10)` |
| `[]` | instructions moteur (non héritées) | `[mode:tem]`, `[*1/2]` |
| `@` | directive top-level | `@sound`, `@actor`, `@template`, `@alphabet` |

**Règle dominante** : `.` = pointer, `:` = lier. Les deux ne se confondent
jamais. Le `*` en position de sujet d'une affectation n'entre pas en conflit
avec `[*N]` (entre crochets) ni avec l'homomorphisme futur `* (= X)` (côté
droit d'une règle) — contextes parser disjoints.

### 6. Harmonisation `@actor` — passage de `:` à `.` pour les références

**v0.7 (à abandonner)** :
```
@actor sitar alphabet:sargam tuning:sargam_22shruti transport:midi(ch:3, vel:100)
```

**v0.8** :
```
@actor sitar
  alphabet.sargam
  tuning.sargam_22shruti
  transport.midi(ch:3, vel:100)
```

Raison : `sargam`, `sargam_22shruti`, `midi` sont des **références à des entités**
(libs / types de transport), pas des affectations. Convention uniforme.

### 7. Format des libs externes

Une lib `lib/sounds/X.json` peut contenir n'importe lequel de ces éléments :

```json
{
  "defaults": { "dur": 500, "alphaMin": 80 },
  "named": {
    "bell_short": { "sample": "bell.wav", "dur": 400 },
    "drum_kick":  { "sample": "kick.wav", "dur": 200, "breakTempo": true }
  },
  "by_terminal": {
    "Sa": "drum_kick",
    "Re": { "sample": "re.wav" }
  }
}
```

S'aligne sur le pattern existant `lib/sounds/tabla_perc.json` (defaults +
templates + by_terminal + override) tout en intégrant la cascade par propriété.

### 8. Pas de compatibilité ascendante

Les 44 grammaires actuelles seront migrées en bloc via un script automatisé.
La v0.7 n'est plus acceptée par le parser v0.8.

### 9. Portée par acteur intégrée d'emblée

Pas reportée à v0.9. La notation `@actor X Y:sound.Z` (niveau 6 de la cascade)
suffit pour exprimer « la note Y jouée par l'acteur X a tel son spécifique ».

## Plan de migration des 44 grammaires existantes

Transformations à appliquer par script (idempotent, mode verify-only
disponible). Audit préalable obligatoire pour identifier les cas atypiques.

| # | Avant (v0.7) | Après (v0.8) |
|---|---|---|
| 1 | `alphabet:NAME` (dans `@actor`) | `alphabet.NAME` |
| 2 | `tuning:NAME` (dans `@actor`) | `tuning.NAME` |
| 3 | `transport:TYPE(...)` (dans `@actor`) | `transport.TYPE(...)` |
| 4 | `sounds:NAME` (dans `@actor`) | `sound.NAME` (singulier, dot) |
| 5 | `@templates` (top-level) | `@template` (singulier) |
| 6 | tout le reste | inchangé |

Les `(...)` runtime (`vel:80`, `pan:64`) et les `[...]` moteur (`[*1/2]`,
`[mode:tem]`) ne sont **pas** touchés par la migration.

Vérification post-migration : suite de tests transpiler doit rester verte +
chaque scène doit re-générer un AST équivalent (modulo les changements de
notation).

## Questions ouvertes laissées aux agents porteurs

1. **Forme AST exacte de `SoundAssignmentAST`** : un champ `subject: string | '*'`
   et `target: { kind: 'named-ref', name: string } | { kind: 'inline-props', props: Partial<SoundConfig> }`. À confirmer après lecture des types existants `BPx/src/types/soundConfig.ts`.

2. **Faut-il un champ `templateMode` dans `SceneAST`** alors que la décision
   est que la section est toujours en mode catalogue ? Recommandation : non,
   c'est implicite — la présence de `@template` = mode catalogue. Simplifie.

3. **Le booléen nu dans bloc inline** (`{ breakTempo, contBeg }` ≡ `{ breakTempo:true, contBeg:true }`) : tokenizer doit promouvoir nom nu → `name:true` au parse. Pas d'impact AST.

4. **`@sound.libname` (lib externe en notation pointée)** vs `@sound name @import "..."` : la convention `@type.libname` est validée pour `@alphabet.X` etc. Pour `@sound`, idem : `@sound.tabla_classical` charge `lib/sounds/tabla_classical.json` et fournit les sons nommés/défauts qu'elle contient au scope scène.

5. **Variantes `:` après notation pointée** : `@alphabet.tabla:transport` accepte le suffixe variante. Cohérent pour `@sound.tabla_classical:simplified` si on a plusieurs variantes d'une même lib. À documenter mais pas à implémenter en v0.8 si pas de cas concret.
