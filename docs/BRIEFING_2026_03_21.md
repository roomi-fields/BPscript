### Nouveaux formats de ratio (3 formats universels)

Tous les fichiers de tempérament/tuning acceptent :
- Fraction : `"9/8"` → exact
- Décimal : `1.05946` → approximation
- Cents : `"100c"` → converti en `2^(cents/1200)`

### Tempéraments paramétriques (Dynamic Tonality)

7 tempéraments à `"type": "parametric"` avec period + generator + mapping.
Le generator est un paramètre continu (pilotable par CV en temps réel).
Familles : meantone, pajara, magic, miracle, porcupine, superpyth, bohlen-pierce.

## Code modifié/créé

### src/dispatcher/ratios.js — NOUVEAU

Normaliseur de ratios. Fonctions :
- `normalizeRatio(value)` — fraction/cents/décimal → float
- `normalizeRatios(array)` — normalise un tableau
- `normalizeTemperament(obj)` — normalise les ratios d'un tempérament
- `normalizeTuning(obj)` — normalise les altérations d'un tuning

**Testé** : 38 tests passés (`test/test_ratios.js`)

### src/dispatcher/resolver.js — RÉÉCRIT

Ancien resolver (163 lignes, hardcodé 12-TET) remplacé par le nouveau (230 lignes, 5 couches, 2 modes).

**Constructeur** : `new Resolver({ alphabet, octaves, tuning, temperament })`
- Instanciable (un par acteur, plus de singleton)
- Mode table : lookup dans ratios[]
- Mode paramétrique : calcul depuis period + generator + mapping
- `setGenerator(cents)` pour morph Dynamic Tonality en temps réel
- `setReference(hz)` pour changer le diapason
- `resolve(token, direction)` → `{ frequency, noteName, alteration, register }`
- Parse toutes les conventions d'octave (suffix, prefix, séparateurs variables)
- Parse les altérations (suffix avec/sans séparateur)
- Support des tunings directionnels (ascending ≠ descending)

**Testé** : 39 tests passés (`test/test_resolver.js`) couvrant :
- Western 12-TET (A4=440, C4=261.63, C#4, Bb4)
- Sargam 22-shruti (fractions exactes, arrows _v/_^)
- Saptak prefix (mandra/madhya/taar)
- Bhairav directionnel (ascending ≠ descending)
- Meantone paramétrique (Dynamic Tonality, setGenerator)
- Bohlen-Pierce (period=3, tritave)
- Gamelan pelog (period=2.02, octave étirée, prefix gamelan)

## Concept d'acteur (@actor)

### Design (dans les docs, pas encore codé)

Un acteur lie : **alphabet + tuning + octaves + transport + eval**

```
@actor sitar  alphabet:sargam  tuning:sargam_22shruti  octaves:saptak  transport:webaudio
@actor tabla  alphabet:tabla_bols  transport:midi(ch:10)
```

Symboles qualifiés par `:acteur` : `Sa:sitar`, résolution implicite si non ambigu.

### EBNF mis à jour
- `actor_directive` ajoutée dans la `scene`
- `actor_props` et `actor_value` définis
- `symbol` et `symbol_call` avec `[ ":" , IDENT ]` optionnel
- `@actor` dans les clés réservées de `@`

### AST mis à jour
- `Scene.actors: ActorDirective[]`
- `ActorDirective { name, properties: { alphabet, tuning, octaves, transport, eval } }`
- `TransportRef { key, params }`
- `Declaration.actor` et `Declaration.runtime` (dual support legacy)
- `Symbol.actor: string | null` et `SymbolCall.actor: string | null`
- Phase "Actor resolver" dans le pipeline AST

### Plan d'implémentation (7 phases)

1. Tokenizer — rien à changer
2. Parser — parse @actor, Symbol.actor (~30 lignes)
3. actorResolver.js — **NOUVEAU fichier** — charge JSON, symbolTable, conflits (~60-100 lignes)
4. Encoder — émet terminalActorMap (~10 lignes)
5. index.js — branche actorResolver dans le pipeline (~5 lignes)
6. Dispatcher — route par acteur via terminalActorMap (~15 lignes)
7. Web UI — un Resolver + Transport par acteur (~30 lignes)

MVP estimé à ~150 lignes (tout explicite, pas de résolution implicite).

## Décisions d'architecture prises

### Pitch
- 5 couches : alphabet → octaves → tempérament → tuning → resolver
- Tout est ratio, multiplication pure, zéro cas spécial
- 3 formats de ratio : fraction, décimal, cents
- period_ratio configurable (2=octave, 3=tritave, 2.02=gamelan)
- Altérations en ratios (pas en steps) → C## ≠ D en just intonation
- Dynamic Tonality : tempéraments paramétriques (period + generator + mapping)
- Gammes composées par tétracordes (jins arabes/turcs)
- Gammes directionnelles (aroha/avaroha ragas)

### Acteur
- Actor = alphabet + tuning + octaves + transport + eval
- Un resolver par acteur (pas global)
- terminalActorMap émise par l'encoder pour le dispatcher
- Résolution implicite si un seul acteur contient le symbole

### Effets / patching
- **Pas de patching dans BPscript** — le runtime (SC, WebAudio) gère le graphe audio
- BPscript pilote les paramètres des effets dans le temps via polymétrie
- Syntaxe : `actor.effect.param(valeur)` en notation dot
- Courbes d'effets = voix parallèles en polymétrie : `{ mélodie , sitar.lpf.cutoff(ramp(200, 4000)) }`

### REPL adapters (backticks)
- 3 protocoles : stdin/stdout (local), TCP (distant), WebSocket (browser)
- Timing best-effort, pas sample-accurate
- getValue() timeout 100ms, fallback dernière valeur
- Erreurs → skip + warning, jamais d'arrêt de scène
- Sessions par clé d'eval (partagées entre acteurs)
- Browser : JS natif, WASM (Pyodide), ou WebSocket vers serveur

### Intégrations
- VCV Rack : résolu par OSC existant
- Ableton : séquence OSC bridge → Ableton Link → dog-fooding → M4L
- on_fail : item dev, traduction mécanique vers _goto/_failed BP3
